# 주행 개발 계획서 (프로젝트 2단계)

## 0. 범위와 전제

이 문서는 프로젝트 4단계 중 2번(주행 개발)만 다룬다.
- 1번 실험(동역학 식별)은 별도 계획서로 완료. 이 계획은 그 실험 결과(정지거리, 최소 회전반경, weight-속도 매핑, yaw 축)가 나온 것을 전제로 파라미터를 채운다.
- 3번 온디바이스 LLM, 4번 HILS는 범위 밖이다. 단, 나중에 붙일 수 있도록 인터페이스(데이터 출력 지점)만 남겨둔다. 구체적으로 Scenario 1의 정찰 로그 출력과 Scenario 2의 실시간 상태 피드가 LLM 입력이 될 자리다.

목표: 고전 알고리즘만으로 두 시나리오의 자율 주행을 완성한다. 강화학습은 이후 단계에서 모듈 단위로 교체한다.

## 1. 정보 경계 정의 (가장 먼저 확정할 것)

자율주행의 의미는 모르는 것을 센서로 알아내는 데 있다. 따라서 무엇을 "안다"로 두고 무엇을 "모른다"로 둘지의 경계가 곧 프로젝트 정의다. map 파일 분석 결과를 근거로 아래와 같이 정한다.

### 1.1 map 파일 사양 (분석 결과)
- 구조: `terrainIndex` (지형 번호) + `obstacles` 배열. 각 obstacle은 prefabName, position(x,y,z), rotation(quaternion).
- 세 맵 모두 terrainIndex 5 (동일 험지).
- notree.map: obstacles 비어 있음 (베이스 험지).
- 현재 정적/MFD 기본 맵 finalmap.map: obstacles 320개. 비교용 mission_map.map: 1204개. 과거 recon_map.map은 1160개로 남아 있으나 기본 정적 맵 경로에서는 사용하지 않는다.
- prefab 종류: Tree(나무), Rock(바위), House001/House002(건물), Human(사람), Car(차량).
- 맵 범위: x 약 1.7~299.9, z 약 1.0~297.9 (대략 300x300 평면), y(높이) 약 6.8~23.9.
- 지형의 높이맵과 강 위치는 파일에 없다. terrainIndex가 시뮬레이터 내장 지형을 가리킬 뿐이다. 강/언덕/바위지대는 terrain 5에 고정되어 렌더링된다.

### 1.2 안다 / 모른다 경계
안다 (사전정보로 코드에 주입). 드론 정찰로 확인한 것은 딱 두 가지다:
- 목적지 좌표. map에 없으므로 코드 상수로 정의 (또는 /set_destination 활용).
- 정찰 루트 A, B의 경유점 시퀀스. 주의: 시뮬레이터에 루트를 입력하는 API가 없다(/set_destination은 목적지 하나뿐). 따라서 "드론이 준 루트 2개"는 실제로는 우리가 경유점 좌표 시퀀스로 코드에 정의하는 것이다. 드론 정찰은 내러티브고, 구현상은 하드코딩된 경유점이다. 맵 스크린샷상 루트 2개가 강 왼쪽 영역에 명확히 보이므로 그 경로를 좌표로 옮긴다.

모른다 (센서로 검출 / 추정 대상):
- 모든 장애물 위치 (나무, 바위, 벽 등). 라이다로 검출. 이것이 인지의 핵심.
- 초소 위치. 시야각을 모르므로 위치도 모르는 것으로 둔다. 정찰 중 카메라(YOLO 건물 식별)나 라이다로 발견하는 대상이다. 초소 발견 자체가 정찰 성과 중 하나.
- 초소 시야각. 값 자체를 주행 알고리즘에 주지 않는다 (6장).
- 지형 상세 (높이/강 위치). 다만 강은 주어진 루트가 강 왼쪽 안전 영역을 지나가므로, 루트를 따르는 한 명시적으로 알 필요가 없다 (3.1, 3.2 참고). 시뮬레이터에서 지형 데이터를 뽑을 수 있는지는 미확인이며, 가능 여부에 따라 분기한다.
- Scenario 2의 돌발 장애물과 적 전차 위치. 실시간 센서로 검출.

### 1.3 구현 원칙
- map 파일을 주행 알고리즘이 통째로 읽으면 안 된다 (장애물 위치를 그대로 아는 것은 인지 우회). 주행 알고리즘의 입력은 목적지와 루트 경유점뿐이다.
- House002 좌표 추출은 사라지지 않고 용도가 바뀐다. 주행 알고리즘에는 주지 않고, 발각 판정 심판 모듈(6장)에만 ground truth로 준다. recon_map 기준 House002는 5개 존재(x/z: 151.9/273, 177.1/169.9, 25.8/262.4, 188.6/79.9, 98.5/165.7). 심판이 어느 초소를 임무 대상으로 삼을지는 지정해야 한다. 같은 좌표를 주행은 모르고 심판만 아는 구조다.
- 전체 obstacles(ground truth)는 버리지 말고 평가용 정답지로 따로 보관한다. 라이다로 검출한 장애물이 실제와 얼마나 맞는지 정량 평가하는 데 쓴다. 공짜 평가 데이터다.

## 2. 좌표계 / 맵 검증 (개발 착수 전 1회)

planner가 전부 좌표 위에서 도니, 좌표계가 틀리면 모든 것이 틀어진다. 코딩 전에 확정한다.
- /info의 playerPos, /update_obstacle의 bbox, /set_destination의 좌표, map 파일의 position이 모두 같은 프레임인지 확인한다. (현재 추정: x/z가 지면 평면, y가 높이. y 범위가 6.8~23.9로 일관되어 같은 프레임으로 보이나 미검증.)
- 검증 방법: 알려진 House002 좌표 위로 전차를 보내 /info의 playerPos가 그 근방을 가리키는지, 같은 위치에서 라이다가 그 건물을 잡는지 대조한다.
- 맵 경계(약 0~300)를 격자 해상도와 함께 확정한다. 예: 1좌표 = 1셀이면 300x300 grid.

## 3. 아키텍처

overview의 레이어 구성을 따른다. 이 단계에서 구현하는 것은 인지/Global/Local 세 층이다.

```
LiDAR / Camera --> [인지] --> 장애물 맵(occupancy) + 객체
                                  |
목적지/루트/초소좌표 ----------> [Global Planning] A* + 경로 평활화
                                  |
                              참조 경로
                                  |
LiDAR ------------------------> [Local Control] Pure Pursuit + APF --> /get_action 명령
                                  |
                              (로그 출력) --> [3번 LLM 자리, stub]
```

### 3.1 인지 (Perception)
- 1차 입력은 라이다다. lidarPoints에서 isDetected=true인 점들을 클러스터링해 장애물 위치로 변환하고, 전차 주변 occupancy grid(또는 코스트맵)를 만든다.
- 험지 주의: 라이다가 지면을 때릴 수 있다(문서 경고). "장애물이냐 지형이냐"를 구분해야 한다. 채널 높이/각도로 거르거나, 일정 높이 이상 점만 장애물로 취급하는 식. 이 terrain/obstacle 분리가 험지 인지의 핵심 난제다.
- 강/물: terrain에 고정된 통행불가 영역이지만 obstacles로도 라이다로도 깔끔히 안 잡힐 수 있다. 결정 필요 - (a) 드론 러프맵 일부로 안다고 두고 통행불가 영역을 사전 마스크로 코스트맵에 박거나, (b) 라이다로 검출 시도. (a)가 현실적이고 단순하다.
- 카메라/YOLO: 적·초소·중요 객체 식별용. 주행 경로 계획의 1차 입력은 아니다. Scenario 2에서 적 전차 식별, Scenario 1에서 초소 확인 등 의미 인지에 쓴다. 거리는 스테레오로 보강 가능하나 우선순위 낮음.
- 센서 퓨전: 라이다(위치/거리) + 카메라(클래스)를 합치는 것은 적/초소 식별이 필요할 때만. 순수 장애물 회피에는 라이다만으로 충분하다. 처음부터 퓨전하지 말고 라이다 단독으로 회피를 완성한 뒤 필요한 곳에만 카메라를 얹는다.

### 3.2 Global Planning
루트가 가는 선이 아니라 전차 폭보다 훨씬 넓은 회랑(corridor)이라는 점이 핵심이다. 따라서 A*는 축소되지 않고 회랑 안에서 본래 역할을 한다. 루트는 "어디까지 가도 되는지"(탐색 공간), A*는 "그 안에서 어떻게 갈지"(최적 경로)로 역할이 분리된다.
- 회랑 정의: 루트 경유점으로 중심선을 만들고 폭(반경)을 줘 회랑 마스크를 생성한다. 폭은 스크린샷의 실제 루트 폭을 좌표로 환산해 정한다. 두 루트는 각각 별도 회랑이다.
- A* on grid: 코스트맵에서 회랑 밖은 통행불가(무한 비용)로 막고, 회랑 안에서만 탐색한다. 회랑 안 코스트맵에 인지로 검출한 장애물, 경사 페널티를 반영해 목적지까지 경로를 찾는다.
- 강 회피가 자동 해결된다: 회랑이 강 왼쪽 안전 영역으로만 그려져 있으므로, 회랑 밖을 탐색하지 않으면 강에 갈 일이 없다. 지형(강 위치)을 몰라도 회랑 경계가 안전 영역을 보장한다. 이로써 3.1에서 고민한 강/통행불가 마스크 문제가 회랑으로 흡수된다. 단 Scenario 2에서 돌발 상황으로 회랑을 크게 이탈하면 보장이 깨지므로, 이탈 시 강 검출 보조가 필요할 수 있다.
- 경로 평활화: A*가 내놓는 격자 지그재그 경로를 직선 구간으로 줄인다 (overview의 LoS 평활화). 이때의 LoS(line-of-sight 가시성 체크로 불필요 경유점 제거)는 6장 초소 시야의 LoS와 용어만 같고 다른 개념이니 혼동 주의.
- 비용 설계는 프로젝트 목표에 종속된다. Scenario 1은 정찰이므로 커버리지/안전 위주, Scenario 2는 진격이므로 도달 시간 + 위협 회피 가중. 목표를 먼저 정해야 비용이 정해진다.

### 3.3 Local Control
- Pure Pursuit로 참조 경로를 추종. lookahead 거리는 1번 실험의 정지거리/최소 회전반경으로 튜닝한다.
- APF(artificial potential field)로 실시간 회피. 라이다가 잡은 가까운 장애물에 반발력, 목표 방향에 인력. Scenario 2의 돌발 장애물과 적 전차 회피가 여기 담당.
- 출력은 /get_action의 moveWS(W/S) + moveAD(A/D) + weight로 매핑. 전차식 조향(회전 후 전진)이라 heading 오차 -> A/D, 전방 거리 -> W로 변환. APF가 강하면 DWA로 교체 검토 가능(명령 구조상 DWA도 잘 맞음).
- APF의 알려진 함정: 지역 최소점(반발력과 인력이 상쇄되어 멈춤). 험지 + 다수 장애물에서 발생 가능. global 경로가 잡아주므로 완화되지만, 갇힘 감지 + 탈출 로직을 두는 게 안전하다.

## 4. Scenario 1: 상세 정찰

목적은 회피가 아니라 정찰과 측정이다. 사용 맵 recon_map.

### 4.1 주행
- 드론이 준 루트 A, B를 순차 주행. 각 루트는 경유점 시퀀스로 정의하고 Global+Local로 따라간다.
- 주행 중 라이다로 미지의 정적 장애물(바위 등)을 검출하고 위치를 기록한다 (정밀 탐색). 검출 결과를 평가용 GT와 대조해 인지 정확도를 측정한다.

### 4.2 로깅 (3번 LLM 입력 자리)
주행하며 아래를 시간축으로 기록한다. 이것이 LLM의 위험 평가 입력이 된다.
- 초소(House002) 시야 범위 내 체류 시간, 실제 발각 횟수 (발각 판정은 6장).
- 지형 험도 (경사, 속도 저하 등 1번 실험 지표 활용).
- 기상/조명 등 환경 값 (lux 등 가용 시).
- 루트별 누적 위험 로그.

이 단계에서 주행 코드는 발각을 피하려 하지 않는다. 그냥 주행하며 발각 여부를 관측해 기록만 한다. 루트 선택(위험 평가)은 3번 LLM의 일이다.

## 5. Scenario 2: 본대 진격

사용 맵 mission_map. Scenario 1에서 (LLM이) 선정한 루트를 따라 목적지(적 기지)로 진격.

### 5.1 주행
- 선정 루트를 Global 참조 경로로 삼아 진격.
- 정찰 때 없던 돌발 장애물 출현 -> 라이다 인지 + APF로 실시간 회피.
- 적 전차 출현 -> 동적 장애물로 취급해 회피. 적은 enemyTracking으로 움직이므로 위치가 변한다. /info의 enemyPos를 쓸지(안다고 가정) 라이다/카메라로 검출할지는 1.2 경계 결정에 따른다. 적도 "모른다"로 두면 센서 검출 필요.

### 5.2 위협 회피
- 초소 시야와 적 전차 사정범위를 위협 영역으로 APF 반발력 또는 코스트맵 비용에 추가한다. Scenario 1과 달리 여기서는 적극 회피한다.
- 단, 시야각/사정범위를 주행 알고리즘이 직접 아는지는 6장 정책에 따른다.

### 5.3 상태 피드 (3번 LLM 입력 자리)
적 위협 수준, 피격 상태, 발각 빈도, 돌발 장애물 우회로 인한 추가 거리 등을 실시간 출력한다. Go/Return 판단은 3번 LLM의 일이며, 이 단계에서는 그 입력 피드만 만든다.

## 6. 위협 영역 처리 (초소 시야 / 적 사정범위)

가장 헷갈리는 부분이라 구조를 명시한다.

### 6.1 책임 분리
- 시야각/사정범위 값은 "환경/심판 모듈"이 가진다. 우리가 임의로 정한 값.
- 주행 알고리즘은 이 값을 모른다 (Scenario 1 기준). 발각됨/안됨, 피격됨/안됨 같은 결과 이벤트만 관측한다.
- 이 분리는 RL에서 보상 함수를 환경이 계산하되 에이전트는 그 함수를 모르는 구조와 같다. 나중에 주행을 RL로 바꿔도 이 경계가 그대로 재사용된다.

### 6.2 발각 판정 모듈 (직접 구현 필요)
- API에 초소 발각 이벤트가 없다. 따라서 별도 모듈을 만든다.
- 입력: 초소 위치(앎) + 우리가 정한 시야각/거리 + 전차 위치(/info playerPos) + 가시선 차단 여부(장애물에 가려지면 발각 안 됨).
- 출력: 매 시점 발각 여부. 이걸 Scenario 1 로그로 보낸다.
- 시야 차단(LoS) 계산: 초소와 전차 사이 직선에 장애물/지형이 있으면 미발각. 이때 장애물 정보는 평가용 GT를 써도 된다 (심판 모듈은 ground truth를 봐도 무방, 주행 알고리즘만 안 보면 됨).

### 6.3 단계별 적용
- Scenario 1: 시야각 모름. 판정 모듈이 발각만 기록. 주행은 회피 안 함.
- Scenario 2: 위협 영역을 APF/코스트맵에 반영해 회피. 이때 시야각/사정범위를 알고 회피할지(현실적으로는 1단계에서 추정된 값을 쓰는 게 자연스러움), 아니면 보수적 고정 반경으로 회피할지 결정 필요.

---

## 7. 단계적 빌드업 (중요)

험지 + 무인지 + 위협 회피를 처음부터 한꺼번에 켜면 무엇이 틀렸는지 분리가 안 된다. 최종 무대가 험지라는 것과 개발을 험지에서 시작하는 것은 다르다. 쉬운 조건에서 올라간다.

1. 평지(Simple Flat) + ground truth 장애물 + 단일 목적지
   - Global A* + Pure Pursuit가 경로를 따라가는지. 인지/회피/위협 다 끄고 파이프라인 검증.
2. 평지 + 라이다 인지 + ground truth 비교
   - 인지가 GT와 얼마나 맞는지. APF 회피 추가. 정적 장애물 회피 완성.
3. 평지 + 위협 영역(고정 반경)
   - APF 위협 반발력, 발각 판정 모듈 검증.
4. 험지(terrain 5) + ground truth
   - 경사/강 코스트맵, 경로 비용 험지 대응. 인지는 아직 GT로.
5. 험지 + 라이다 인지
   - terrain/obstacle 분리 난제 본격 대응. 험지 인지 정확도 측정.
6. 험지 + recon_map 전체 (Scenario 1 통합)
   - 루트 A/B 주행 + 정찰 로깅 + 발각 판정 전체 연동.
7. 험지 + mission_map (Scenario 2 통합)
   - 돌발 장애물 + 적 전차 동적 회피 + 상태 피드.

각 단계는 다음으로 넘어가기 전에 성공 기준(9장)을 통과해야 한다.

---

## 8. 작업 순서 (코딩 단계)

1. 좌표계/맵 검증 (2장).
2. 셋업 스크립트: recon_map 파싱 -> House002 좌표 추출, 평가용 GT obstacles 저장.
3. 코스트맵 모듈: grid 정의, 장애물/경사/통행불가 반영.
4. Global: A* + 경로 평활화.
5. Local: Pure Pursuit (회피 없이 경로 추종부터). 1번 실험값으로 lookahead 튜닝.
6. 1~4단계(7장) 까지 평지에서 검증.
7. 인지 모듈: 라이다 클러스터링 -> occupancy. GT 대조 평가.
8. APF 회피 추가, 위협 반발력, 발각 판정 모듈.
9. 험지로 전환, terrain/obstacle 분리 대응 (4,5단계).
10. Scenario 1 통합 (6단계) + 정찰 로그 출력 인터페이스.
11. Scenario 2 통합 (7단계) + 상태 피드 인터페이스.

---

## 9. 검증 / 지표

- 도달 성공률: 목적지 도달 비율 (충돌/이탈 없이).
- 충돌 횟수: /collision 이벤트 수.
- 인지 정확도: 라이다 검출 장애물 vs 평가용 GT (위치 오차, 미검출/오검출률).
- 경로 효율: 경로 길이 / 직선거리, 주행 시간.
- 발각 지표(Scenario 1): 초소 시야 체류시간, 발각 횟수 (로그 정상 산출 여부).
- 회피 지표(Scenario 2): 돌발 장애물 회피 성공률, 우회 추가거리, 피격 횟수.

각 빌드업 단계의 통과 기준으로 사용한다. 예: 1단계는 충돌 0 + 도달 성공, 2단계는 인지 미검출률 기준치 이하.

## 10. 리스크 / 미결정 사항

미결정 (착수 전 정해야 함):
- 임무 대상 House002 지정 (recon_map에 5개).
- 적 전차 위치를 안다(enemyPos)로 둘지 센서 검출로 둘지.
- 강/통행불가 영역을 사전 마스크로 줄지 라이다 검출할지 (사전 마스크 권장).
- Scenario 2에서 위협 반경을 알고 회피할지 보수적 고정값으로 회피할지.
- 코스트맵 비용 설계 방향 (정찰=안전/커버리지, 진격=시간+위협). 프로젝트 목표에 종속.

리스크:
- 험지 라이다의 terrain/obstacle 분리가 이 단계 최대 난제. 평지 인지보다 훨씬 까다롭다. 7장 단계적 접근으로 분리해서 다룬다.
- APF 지역 최소점. 갇힘 감지/탈출 로직 필요.
- map의 position이 다른 API와 같은 좌표계라는 보장이 아직 없다 (2장에서 검증 전까지 가정).
- finalmap(320)과 mission_map(1204)의 obstacle 차이가 "돌발 장애물"인지 확인 필요. 현재 production 인식/fusion class는 rock/person/tank/car/house이며, 그 밖의 prefab은 정적 비교/표시에서 unknown으로 취급된다.
- 1번 실험이 끝나지 않으면 Local 파라미터(lookahead 등)를 임시값으로 둘 수밖에 없다. 실험 완료가 선행되어야 정밀 튜닝 가능.

범위 밖 (이 단계에서 안 함):
- 3번 온디바이스 LLM (위험 평가, Go/Return). 입력 인터페이스만 남김.
- 4번 HILS. 센서 고장/하드웨어 이벤트 주입은 추후.
- 강화학습. 주행 안정화 후 모듈 단위 교체.
- 사격(fire), 포탑 제어. 주행과 무관.
