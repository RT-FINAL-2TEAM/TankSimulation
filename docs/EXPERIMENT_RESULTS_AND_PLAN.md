# 평지 주행 동역학 시스템 식별 실험 결과 (E1~E6)

본 문서는 `tests/step0_dynamics/data/experiment_log_flat.csv`를 기반으로 산출된 전차 시뮬레이터의 험지 외(평지) 주행 특성을 분석한 결과입니다.

## 1. 실험 개요
- **목적**: 자율 주행 계획(Planning) 모듈이 활용할 전차의 기구학 및 동역학적 파라미터 확보
- **수집 데이터**: 16개 Trial, 총 약 13분의 주행 로그
- **분석 스크립트**: `scripts/analyze_dynamics.py`

## 2. E1: 축 식별 (Yaw Axis)
제자리 회전 시 Euler 각도 변화량을 분산 분석한 결과, 가장 크게 변동하는 회전 축은 다음과 같이 식별되었습니다.
- **Yaw 대응 축**: `body_x`

## 3. E2 & E3: 정상 상태(Steady-State) 응답
입력 가중치(`weight`)에 따른 최고 속도 및 최대 회전 각속도 산출 결과입니다. 두 항목 모두 입력값에 대해 매우 선형적인 비례 관계를 보입니다.

### E2: 전진 속도 (w_ws)
| w_ws (입력) | 정상 속도 (m/s) |
|-------------|-----------------|
| 0.1 | 1.95 |
| 0.3 | 5.82 |
| 0.5 | 9.60 |
| 0.8 | 15.46 |
| 1.0 | 19.45 |
> [!NOTE]
> 전진 속도는 $V_{ss} \approx 19.45 \times w_{ws}$ 로 모델링할 수 있습니다.

### E3: 회전 속도 (w_ad)
| w_ad (입력) | 정상 Yaw Rate (deg/s) |
|-------------|-----------------------|
| 0.1 | 4.01 |
| 0.3 | 12.00 |
| 0.5 | 20.16 |
| 0.8 | 32.00 |
| 1.0 | 38.84 |
> [!NOTE]
> 회전 속도는 $\dot{\psi}_{ss} \approx 38.84 \times w_{ad}$ 로 모델링할 수 있습니다.

## 4. E4 & E5: 과도 응답 및 제동
가속 지연과 정지 관성을 확인하기 위한 항목입니다.

- **E4 가속 시정수(Time Constant)**: `4.69초`
  - 정지 상태에서 풀액셀(`w_ws=1.0`) 인가 시, 최고 속도의 63.2%(약 12.3m/s)에 도달하기까지 걸리는 시간입니다. 상당히 무거운 관성 특성을 가집니다.
- **E5 정지 특성**:
  - `w_ws=1.0` 상태에서 정지(STOP) 명령 하달 시 완전히 멈출 때까지의 거리: **9.26m**
  - 소요 시간: **1.55초** (실제 속도가 0.1m/s 이하로 수렴할 때까지의 실 제동 시간)
> [!WARNING]
> 최고 속도(약 19.45m/s) 주행 중 급정거 시 약 9.26m의 제동 거리가 발생합니다. 내비게이션 알고리즘 설계 시 목표 지점 도착 전 최소 10m 전부터 사전 제동(브레이킹) 로직을 적용해야 안전합니다.

## 5. E6: 회전 반경 기하
곡선 주행 시 전진(w_ws)과 회전(w_ad)이 동시에 인가될 때 형성되는 회전 반경($R$)입니다.

| w_ws | w_ad | 속도 (m/s) | Yaw Rate (rad/s) | 회전 반경 (m) |
|------|------|------------|------------------|---------------|
| 0.5 | 0.5 | 9.8 | 0.35 | **27.95** |
| 0.8 | 0.3 | 15.4 | 0.20 | **74.90** |
| 0.3 | 0.7 | 5.8 | 0.48 | **11.90** |

> [!TIP]
> 입력 비율에 따라 회전 반경이 $11m \sim 75m$ 수준으로 달라집니다. 협소한 구역에서는 조향 비중(`w_ad`)을 극대화해야 합니다.

## 6. E7: 험지(Rough Terrain) 비교 분석
`tests/step0_dynamics/data/experiment_log_rough.csv`(험지) 데이터와 평지 데이터를 비교 분석한 결과입니다.

### 동역학 외란 (속도/조향/제동)
- **속도 저하율**: `1.000` (평지 대비 100%)
- **조향(Yaw Rate) 변화율**: `1.014` (평지 대비 약 101%)
- **제동 거리 비율**: `0.977` (평지에 비해 제동 거리가 약 2.3% 짧아짐)
> [!TIP]
> 속도 저하가 발생하지 않고 조향 성능도 거의 일정하게 유지되는 것으로 보아, 본 시뮬레이터의 전차 구동은 지형 마찰이나 경사 저항을 크게 받지 않는 **아케이드 성향(스크립트 구동)**으로 구현되어 있습니다. 따라서 Planning 시 험지에서도 평지와 동일한 속도/회전 모델을 적용해도 무방합니다.

### 차체 자세 외란 (Pitch / Roll)
직진 주행(E2) 중 차체 자세각(`body_y`, `body_z`)의 분산을 측정한 결과입니다.
- **평지 분산 (Y / Z)**: `0.00 / 0.00`
- **험지 분산 (Y / Z)**: `20.67 / 13.50` (표준편차 약 4.5도 / 3.7도, 최대 변동 폭 약 ±15~30도)
> [!NOTE]
> 평지에서는 수평을 유지하고, 험지에서는 지형 굴곡을 따라 Pitch/Roll이 변동합니다. 표준편차 기준 약 4~5도 수준으로, 멀미 수준의 극단적 요동은 아닙니다. 다만 정밀한 객체 탐지나 스테레오 거리 측정에는 영향을 줄 수 있으므로, 필요 시 소프트웨어적인 Pitch/Roll 보정을 고려할 수 있습니다.
> [!WARNING]
> 정정 이력: 이전 분석에서 이 분산이 `31490 / 11393`으로 나온 것은 각도를 0~360 범위 그대로 두고 분산을 계산해, 0/360 경계를 오가는 샘플이 값을 약 1500배 부풀린 오류였습니다. Pitch/Roll은 0 기준 작은 각도이므로 -180~180으로 정규화한 뒤 재계산한 값이 위 수치입니다.

- **파라미터 파일**: `configs/tank_parameters.yaml`

# 전차 시뮬레이터 동역학 식별 실험 — 작업 계획서

## 0. 한 줄 요약

소스 코드도 물리 파라미터 문서도 없는 시뮬레이터를 대상으로, HTTP End Point 응답만으로 전차를 구동하고 `/info` 로그를 수집하여 입력(이산 명령 + weight) 대 출력(속도/위치/자세)의 관계를 측정한다. 이것은 본질적으로 블랙박스 시스템 식별이다.

---

## 1. 배경과 전제

### 1.1 통신 구조 (중요)
- 시뮬레이터가 HTTP 클라이언트, 우리 코드가 서버다. 시뮬레이터가 Request Interval 주기로 우리 엔드포인트를 호출하고, 우리는 응답만 한다.
- "명령을 push하는" API는 없다. 시뮬레이터가 물어볼 때 무엇을 돌려주느냐로 전부 제어한다.
- 이 구조 덕분에 trial 진행/리셋/시작/명령주입을 전부 코드로 자동화할 수 있다. 사람이 손대야 하는 부분은 지형 선택뿐이다.

### 1.2 코드로 되는 것 / 안 되는 것
코드로 됨:
- 전차 구동 — `/get_action` 응답 (W/S, A/D, weight)
- 상태 수집 — `/info` (position, speed, body 각도 3축, time 등)
- 에피소드 시작 — 일시정지 상태면 시뮬레이터가 `/start`를 1초마다 폴링, 여기서 `control: "start"` 반환
- 리셋 — `/info` 응답에 `control: "reset"`, 리셋되면 `/init` 재호출됨
- trial별 시작 좌표 — `/init` 응답의 blStartX/Y/Z

코드로 안 됨 (UI 수동):
- 지형 변경 (Terrain Settings 메뉴)
- 맵 로드 (File > Load Map)
- 장애물 배치 (Object Edit Mode) — `/update_obstacle`는 시뮬레이터가 우리에게 정보를 보내주는 방향이지 우리가 놓는 게 아님

### 1.3 미확정 사항 (이 실험으로 밝혀낼 것)
아래는 문서/샘플코드에 명시가 없어 실측으로 확정해야 한다. 추측은 추측으로 표기.
- BodyX/Y/Z 중 어느 축이 yaw(heading)인지. (추측: Y가 yaw일 가능성이 높지만 미확인. E1에서 확정)
- "기존 물리량 * weight"에서 weight가 속도/힘/가속 중 무엇을 스케일하는지. (E2/E4에서 확정)
- 회전(A/D)이 제자리 회전인지 호를 그리는지, 최소 회전반경이 얼마인지. (E6에서 확정)
- pitch/roll을 BodyX/Z로 읽을 수 있는지. (추측: 가능할 것으로 보이나 험지 E에서 검증)
- 좌표 단위와 Distance(가상 km 환산)의 관계. (부차적, 필요시 확인)

## 2. 시뮬레이터 연동 및 설정 (실험 전)

### 2.1 연동 절차
시뮬레이터가 HTTP 클라이언트, 우리 Flask 서버가 End Point다. 시뮬레이터가 Request Interval마다 우리 포트로 요청을 보내고, 서버는 켜두기만 하면 된다. 순서:
1. Flask 서버 실행. 샘플 Player 코드 기반, port 5000. (서버를 먼저 띄워두면 시뮬레이터 연결이 매끄럽다.)
2. 시뮬레이터 실행. Setting > Properties > Server Setting에서 Mode = Simulation(적과 교전 환경, Versus 아님), Request Port = 5000, Request Interval 설정.
3. 모드 활성화. 이 실험에는 trackingMode True(전차를 /get_action으로 제어)와 logMode True(/info로 데이터 수집)가 필수다. Run 메뉴에서 수동으로 켤 수도 있으나, /init 응답으로 켜는 것이 코드 기반 자동화에 맞다 (2.2 참고).
4. 씬이 시작되면 /init이 호출되어 서버가 초기 설정을 반환한다. Start로 에피소드가 진행되면 /get_action, /info 등이 Request Interval 주기로 호출되기 시작한다.
5. 연결 확인. 서버 콘솔에 요청 로그(/init, /get_action, /info)가 찍히면 연동된 것이다. 안 찍히면 포트 번호와 방화벽을 확인한다.

Enemy 서버(port 5100)는 이 실험에 불필요하다. enemyTracking을 끄므로 적을 구동할 필요가 없다.

### 2.2 필수 변경
- logMode를 반드시 True로 설정한다. 샘플 `/init`에는 False로 되어 있어 그대로 두면 `/info`가 호출되지 않아 데이터 수집이 안 됨. 이번 실험의 핵심 데이터 소스가 `/info`이므로 필수.
- enemyTracking은 False 유지. 적 전차가 따라오면 측정에 간섭한다. (샘플 기본값 False라 OK)
- trackingMode True 유지. 키보드 입력이 막히고 `/get_action`으로만 구동되어 자동화에 유리하다.

### 2.3 Request Interval
- 짧을수록 `/info` 샘플 해상도가 올라가 과도응답(가속/감속 곡선)을 잘 잡는다.
- 단 문서 경고대로 너무 짧으면 통신 부하로 시뮬레이터 성능에 영향. 적당한 값에서 시작해 과도응답이 너무 거칠면 줄인다.
- 샘플링 주기 = Request Interval이라는 점을 데이터 해석 시 항상 의식한다.

### 2.4 지형
- 평지: Simple Flat (문서가 테스트용으로 명시). 경사 외란이 없어 순수 동역학 측정용.
- 험지: 실제로 사용할 특정 맵 하나. 같은 명령이 경사에서 얼마나 어긋나는지(외란) 측정용.
- 지형 변경 시 장애물과 전차 상태가 초기화되므로, 지형을 먼저 고정하고 그 안에서 trial을 돌린다.

## 3. 자동화 하네스 설계

### 3.1 핵심 아이디어
샘플 `/get_action`은 `combined_commands` 리스트에서 `pop(0)`으로 명령을 하나씩 소비한다. 이 구조를 그대로 활용해, 각 trial을 "명령 시퀀스(큐)"로 정의한다. 서버는 trial 인덱스와 현재 큐를 상태로 들고 있는 상태머신이 된다.

### 3.2 엔드포인트별 역할
- `/init` (GET): 현재 trial의 시작 좌표(blStartX/Y/Z)와 startMode를 반환. trial마다 좌표를 바꿔 위치를 통제.
- `/start` (GET): 시작할 준비가 되면 `control: "start"`, 아직이면 `""`(또는 pause) 반환.
- `/get_action` (POST): 현재 trial 큐에서 다음 명령을 pop해서 반환. 큐가 비면 STOP 반환.
- `/info` (POST): 매 호출마다 (time, position, speed, body 각도)를 CSV에 로깅. trial 종료조건을 만족하면 다음 호출에서 `control: "reset"`을 반환해 다음 trial로 넘어감.

### 3.3 trial 종료조건 (둘 중 선택/혼합)
- 시간 기반: `/info`의 time 필드가 trial 제한시간(예: 가속 측정은 짧게, 정지거리는 충분히)을 넘으면 종료. 샘플에 `time > 15` 예시가 주석으로 이미 있음.
- 상태 기반: 정지거리 측정은 speed가 0 근처로 수렴하면 종료, 회전 측정은 목표 yaw 변화량 도달 시 종료.

### 3.4 두 페이즈 trial (heading 통제용)
`/init`으로 시작 좌표는 정하지만 시작 heading은 못 정한다. 특정 방향 출발이 필요한 실험(예: 경사 횡단)은 trial을 두 페이즈로 구성:
1. 정렬 페이즈 — 제자리 회전(A 또는 D)으로 목표 heading까지 회전, `/info`의 yaw로 도달 확인.
2. 측정 페이즈 — 도달 후부터 본 명령 시퀀스 시작, 이 구간만 분석에 사용 (CSV에 페이즈 플래그 기록).

### 3.5 시간 기준
- 모든 시간 분석은 `/info`의 time 필드(sim time)를 쓴다. 벽시계 시간 금지.
- 이유: Frame Rate 설정에 따라 시뮬 시간과 실제 시간이 어긋날 수 있다고 문서에 명시됨. sim time이 진실.

## 4. 실험 항목

평지에서 E1~E6를 먼저 깨끗하게 측정한 뒤, 험지에서 E2~E6를 반복하고 경사 외란(E7)을 추가한다.

### E1. 축 식별 (yaw 찾기)
- 명령: WS=STOP, A를 적당한 weight로 유지 (제자리 회전 시도).
- 측정: BodyX/Y/Z 중 어느 값이 단조 증가/감소하는지.
- 산출: heading에 해당하는 축 확정. 이후 모든 회전 분석의 기준.

### E2. 정상상태 속도 매핑
- 명령: W를 weight 0.1 / 0.3 / 0.5 / 0.8 / 1.0으로 각각 충분히 길게 유지 (속도가 평평해질 때까지).
- 측정: 각 weight에서의 정상상태 speed.
- 산출: weight 대 정상속도 곡선. 선형인지 여부. 명령 직후 speed가 즉시 점프하면 weight가 속도를 스케일, 천천히 차오르면 힘/가속을 스케일한다는 단서.

### E3. 정상상태 회전 매핑
- 명령: WS=STOP, A(또는 D)를 weight 0.1~1.0으로 각각 유지.
- 측정: weight별 yaw rate (E1에서 찾은 축의 시간당 변화).
- 산출: weight 대 회전속도 곡선. 최대 yaw rate.

### E4. 가속 과도응답
- 명령: 정지 상태에서 W weight 1.0 인가.
- 측정: speed가 0에서 정상상태까지 차오르는 곡선 (sim time 기준).
- 산출: 가속 시정수 / 관성 감각. path following 시 명령 반응 지연 추정에 사용.

### E5. 감속 / 정지거리 (제어상 가장 중요)
- 명령: 최고속 주행 중 STOP 인가.
- 측정: STOP 시점부터 speed가 0이 될 때까지의 시간과 이동 거리(position 적분 또는 시작/끝 좌표 차).
- 산출: 정지거리. 이 값이 크면 웨이포인트 오버슈트가 발생하므로 lookahead 거리 / 감속 시작 시점 튜닝에 직결.

### E6. 회전 기하
- 명령: W와 A를 동시 인가 (여러 weight 조합).
- 측정: 이동 경로의 곡률, 최소 회전반경.
- 산출: 제자리 회전형인지 호 주행형인지 확정. 최소 회전반경. local path가 만들 수 있는 곡선의 한계.

### E7. 험지 외란 (험지에서만)
- E2~E6를 험지에서 반복하여 평지 대비 차이를 본다.
- 추가 측정:
  - 오르막에서 같은 W weight일 때 정상속도 저하량.
  - 내리막에서 가속/오버슈트.
  - 경사 횡단 시 측면 밀림(drift) — heading은 유지되는데 위치가 옆으로 새는지.
  - pitch/roll 발생 여부 (BodyX/Z 추정값 변화).
- 주의: 험지는 단일 조건이 아니다. 사용할 맵 한 곳으로 한정하므로 그 맵 안에서 경사가 다른 구간 몇 곳을 골라 trial 시작 좌표로 지정한다.

## 5. 데이터 로깅 및 결과 산출물

### 5.1 CSV 스키마 (`/info` 호출마다 1행)
제안 컬럼:
- trial_id — trial 번호
- phase — align / measure (3.4의 두 페이즈 구분)
- cmd_ws, w_ws — 현재 인가 중인 전후진 명령과 weight
- cmd_ad, w_ad — 현재 인가 중인 회전 명령과 weight
- sim_time — `/info`의 time
- pos_x, pos_y, pos_z — playerPos
- speed — playerSpeed
- body_x, body_y, body_z — playerBodyX/Y/Z
- (선택) terrain — flat / rough, 분석 시 구분용

명령 컬럼은 서버가 현재 trial에서 무엇을 인가 중인지 직접 기록한다 (시뮬레이터가 되돌려주지 않으므로 서버가 자기 상태를 적는다).

### 5.2 파일 분리
- trial별 또는 지형별로 파일을 나눠 저장하면 분석이 편하다.
- 시뮬레이터 자체의 Save Log 기능(`log_data` 폴더)도 켜두면 교차 검증용으로 쓸 수 있으나, 1차 데이터는 우리 서버가 적는 CSV로 한다.

### 5.3 결과 산출물 (측정 후 반드시 남길 것)
측정만 하고 끝내면 안 된다. 정지거리/회전반경 등은 주행 개발에서 계속 참조하므로, 결과를 세 단계로 남긴다. 핵심은 사람용 문서와 코드용 파일을 분리하는 것이다.

1. 측정 원본 CSV (5.1) — 가공 전 raw. 재분석/검증의 근거로 보관.
2. 파라미터 파일 (JSON 또는 YAML) — 코드가 읽는 산출물. 주행 코드가 이 파일을 직접 로드해 lookahead 등을 설정한다. 사람용 산문만 있으면 값을 손으로 옮겨야 하므로 이 파일이 실질적으로 가장 중요하다. 담을 값:
   - yaw에 해당하는 body 축 (E1 결과)
   - weight별 정상속도 테이블 (E2)
   - weight별 yaw rate 테이블 (E3)
   - 가속 시정수 (E4)
   - 정지거리 (E5)
   - 최소 회전반경, 회전 형태(제자리/호) (E6)
   - 험지 보정계수 (경사별 속도 저하율, drift 등 E7)
3. 요약 보고서 (md) — 사람이 읽는 산출물. 각 실험의 결론, 곡선 그래프, weight-속도 관계가 선형인지 등의 해석, 평지 대 험지 비교, 미확정으로 남은 항목.

측정 조건 동봉: 파라미터 파일과 보고서 모두에 측정 조건을 기록한다. 시뮬레이터 버전(현재 2.3.8), Request Interval, 사용한 지형(terrain index), 측정 날짜. 조건이 바뀌면 값도 바뀌므로, 조건 없이 값만 남기면 나중에 재현/신뢰가 안 된다.

## 6. 작업 순서 (코드 작성 단계)

1. &nbsp;샘플 Player 코드 기반으로 서버 골격 잡기. logMode=True로 변경.
2. 상태머신 + trial 큐 구조 구현 (`/init`, `/start`, `/get_action`, `/info` 연동).
3. CSV 로거 구현 (5.1 스키마).
4. 단일 trial 스모크 테스트 — W weight 1.0 짧게 인가 후 reset까지 한 사이클 도는지 확인. 좌표/속도/시간이 CSV에 제대로 쌓이는지 검증.
5. E1 (축 식별) 먼저 실행 — yaw 축을 확정해야 이후 회전 분석이 가능.
6. 평지에서 E2~E6 trial 시퀀스 정의 후 일괄 실행.
7. 험지 맵으로 수동 전환 후 시작 좌표 재확인(6.2 리스크 참고), E2~E7 실행.
8. 분석 노트북 — 곡선 피팅(정상속도, 시정수, 정지거리, 회전반경), 평지 대 험지 비교.

## 7. 리스크 / 주의사항

- 과도응답 해상도가 Request Interval에 묶인다. 가속/감속 곡선이 너무 거칠면 Interval을 줄이되, 너무 줄이면 통신 부하. 절충 필요.
- 험지 시작 좌표가 지형에 박히거나 공중에 뜰 수 있다 (Y값 문제). 험지 trial 전에 좌표별로 1회씩 눈으로 확인하고 안전한 좌표만 사용.
- 시작 heading을 `/init`으로 못 정한다. 방향 의존 실험은 반드시 정렬 페이즈(3.4)를 거친다.
- 시간은 무조건 sim time. 벽시계로 분석하면 Frame Rate 설정에 따라 틀어진다.
- 명령 인가 상태는 시뮬레이터가 echo해주지 않으므로 서버가 자기가 보낸 명령을 직접 CSV에 적어야 한다. 안 그러면 출력만 있고 입력이 없어 분석 불가.
- 좌표 단위가 미확정이다. 정지거리/회전반경을 "좌표 단위"로 먼저 내고, 실제 거리 환산이 필요하면 Distance 표시(가상 km)와 대조해 스케일을 추정.

## 8. 범위 밖 (이번 단계에서 안 하는 것)

- path planning 알고리즘 구현 (global/local) — 동역학 파악 후 별도 단계.
- 강화학습 — RL로 갈 경우 명시적 시스템 식별 자체가 불필요해질 수 있음. 다만 평지·험지 둘 다 학습 분포에 넣어야 한다는 요구는 그대로 남음.
- 객체 탐지(/detect), 스테레오(/stereo_image), 사격(fire) — 주행 동역학과 무관하므로 제외.
